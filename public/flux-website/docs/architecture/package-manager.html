Flux Package Manager

1. Overview

The Flux package manager is integrated into the flux CLI. It handles dependency resolution, package building, publishing, and registry interaction. Design is inspired by Cargo (Rust), npm (Node.js), and Go modules.

2. Registry Architecture

2.1 Official Registry

URL: https://packages.flux-lang.org

The registry provides: - Package metadata (name, version, description, dependencies) - Compressed package archives (.flux-pkg) - Download statistics and popularity metrics - Security advisories - Search API - User authentication (API tokens)

2.2 Package Naming

  • Names: lowercase letters, digits, hyphens, underscores
  • Must start with a letter
  • 3–64 characters
  • Must be unique in the registry
  • Namespace support: @org/package for organization-scoped packages

2.3 Private Registries

# Flux.toml
[registries]
my-company = { url = "https://flux-registry.mycompany.com" }

[dependencies]
internal-lib = { version = "1.0", registry = "my-company" }

3. Dependency Resolution

3.1 Version Constraints

Follows Semantic Versioning:

Syntax Meaning Example
"1.2.3" Compatible ≥1.2.3, <2.0.0
"=1.2.3" Exact Exactly 1.2.3
"~1.2.3" Patch-compatible ≥1.2.3, <1.3.0
">=1.2, <2.0" Range Explicit range
"*" Any Latest

3.2 Resolution Algorithm

  1. Parse: Read all Flux.toml files and build a dependency graph
  2. Collect: For each dependency, fetch available versions from the registry
  3. Resolve: Use a SAT-solver-based algorithm to find a compatible version set
  4. Verify: Check that the resolved set satisfies all constraints
  5. Lock: Write the resolved versions to Flux.lock

3.3 Conflict Resolution

When multiple packages depend on different versions of the same package:

  1. Compatible versions: Select the highest version that satisfies all constraints
  2. Incompatible versions: Error with a clear explanation and suggestion to update constraints
  3. Duplicate dependencies: Only one version of each package is allowed (no diamond dependency issues)

4. Package Sources

Source Flux.toml Syntax
Registry package = "1.0"
Git package = { git = "url", branch = "main" }
Git tag package = { git = "url", tag = "v1.0" }
Git rev package = { git = "url", rev = "abc123" }
Local path package = { path = "../local" }

5. Lock File (Flux.lock)

The lock file records exact resolved versions for reproducible builds:

# This file is auto-generated by flux. Do not edit manually.
# Run `flux update` to refresh.

[[package]]
name = "flux_http"
version = "2.1.4"
source = "registry+https://packages.flux-lang.org"
checksum = "sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
dependencies = [
    "flux_io 1.0.1",
    "flux_async 1.2.0",
]

[[package]]
name = "flux_io"
version = "1.0.1"
source = "registry+https://packages.flux-lang.org"
checksum = "sha256:a1b2c3d4..."

Rules: - Commit Flux.lock to version control for applications - Do NOT commit Flux.lock for libraries (let consumers resolve)

6. Publishing

6.1 Workflow

# 1. Authenticate
flux login

# 2. Verify the package
flux publish --dry-run

# 3. Publish
flux publish

6.2 Pre-Publish Checks

Before publishing, flux publish verifies: - Flux.toml has required fields (name, version, license, description) - All source files compile without errors - Tests pass - Version is not already published - Package size is under 10 MB (configurable) - No local path dependencies

6.3 Yanking

Published versions can be yanked (soft-deleted) but not removed:

flux yank --version 1.2.3
flux yank --version 1.2.3 --undo   # Un-yank

Yanked versions: - Cannot be downloaded by new users - Continue to work for existing Flux.lock files

7. Package Signing

7.1 Signing

Packages are cryptographically signed during publish:

# Generate a signing key
flux key generate

# Sign packages automatically on publish
flux publish  # Uses stored key

# Verify a package signature
flux verify <package> <version>

7.2 Trust Model

  • Publishers sign packages with their private key
  • The registry stores the public key and signature
  • Consumers verify signatures during flux install
  • The flux audit command checks for known vulnerabilities

8. Security

8.1 Supply Chain Security

Feature Description
Checksum verification SHA-256 checksums verified on download
Package signing Ed25519 signatures on all published packages
Audit flux audit checks for known vulnerabilities
Lock file Ensures reproducible builds
Minimal permissions Packages declare required permissions
Source verification Registry verifies package source matches

8.2 Audit

flux audit

# Output:
# ✅ No vulnerabilities found in 42 dependencies
# 
# Or:
# ⚠️  1 vulnerability found:
#   flux_xml 1.2.3 — CVE-2026-1234 (High)
#   XXE vulnerability in XML parsing
#   Fix: update to flux_xml >= 1.2.4

9. Caching

9.1 Global Cache

Downloaded packages are cached in ~/.flux/cache/:

~/.flux/cache/
├── registry/
│   └── packages.flux-lang.org/
│       ├── flux_http-2.1.4.flux-pkg
│       └── flux_json-1.0.2.flux-pkg
├── git/
│   └── github.com-abc123/
└── src/
    └── flux_http-2.1.4/    # Extracted source

9.2 Cache Commands

flux cache list     # Show cached packages
flux cache clean    # Clear the cache
flux cache size     # Show cache size

10. Workspaces

For multi-package projects:

# Root Flux.toml
[workspace]
members = [
    "compiler/flux_lexer",
    "compiler/flux_parser",
    "compiler/flux_ast",
    "cli",
]

[workspace.dependencies]
flux_ast = { path = "compiler/flux_ast" }
serde = "1.0"

[workspace.package]
version = "0.1.0"
edition = "2026"
license = "MIT"

# Build all workspace members
flux build --workspace

# Build a specific member
flux build -p flux_lexer

# Test everything
flux test --workspace

# Run a specific binary
flux run -p cli
