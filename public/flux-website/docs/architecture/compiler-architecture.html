Flux Compiler Architecture

1. Pipeline Overview

Source Code (.flux)
       │
       ▼
┌─────────────┐
│    Lexer     │  → Token Stream
│ (flux_lexer) │
└──────┬──────┘
       ▼
┌─────────────┐
│   Parser     │  → Abstract Syntax Tree (AST)
│(flux_parser) │
└──────┬──────┘
       ▼
┌─────────────┐
│  Name Res.   │  → Resolved AST (symbols linked to declarations)
│ (flux_sema)  │
└──────┬──────┘
       ▼
┌─────────────┐
│ Type Checker │  → Typed AST (every expression has a verified type)
│(flux_types)  │
└──────┬──────┘
       ▼
┌─────────────┐
│  MIR Lower   │  → Mid-Level IR (simplified, desugared)
│ (flux_mir)   │
└──────┬──────┘
       ▼
┌─────────────┐
│  Borrow Ck.  │  → Verified MIR (ownership/borrowing valid)
│ (flux_mir)   │
└──────┬──────┘
       ▼
┌─────────────┐
│  Optimizer   │  → Optimized MIR
│(flux_optim)  │
└──────┬──────┘
       ▼
┌─────────────┐
│  LLVM IR Gen │  → LLVM IR (.ll)
│(flux_codegen)│
└──────┬──────┘
       ▼
┌─────────────┐
│  LLVM        │  → Object File (.o)
│  Backend     │
└──────┬──────┘
       ▼
┌─────────────┐
│   Linker     │  → Executable / Library
└─────────────┘

From To Data

3. Lexer (flux_lexer)

3.1 Responsibilities

  • Convert UTF-8 source text into a stream of tokens
  • Track source locations (file, line, column, byte offset)
  • Handle string interpolation (nested lexing)
  • Error recovery (skip invalid characters, report errors, continue)

3.2 Key Types

pub struct Token {
    pub kind: TokenKind,
    pub span: Span,
    pub text: SmolStr,  // Interned string for identifiers/keywords
}

pub struct Span {
    pub file_id: FileId,
    pub start: u32,   // Byte offset
    pub end: u32,
}

pub enum TokenKind {
    // Keywords
    Let, Mut, Fn, Struct, Class, Enum, Trait, ...
    // Literals
    IntLiteral, FloatLiteral, StringLiteral, ...
    // Operators
    Plus, Minus, Star, Slash, Eq, EqEq, ...
    // Delimiters
    LBrace, RBrace, LParen, RParen, LBracket, RBracket, ...
    // Special
    Eof, Error,
}

3.3 Performance

  • Single-pass, zero-allocation lexing (arena allocator for tokens)
  • SIMD-accelerated whitespace/identifier scanning where possible
  • Lazy string interning

4. Parser (flux_parser)

4.1 Design

  • Recursive descent parser (Pratt parsing for expressions)
  • Produces a concrete syntax tree (CST) that preserves all tokens
  • CST is then lowered to AST (abstract syntax tree) with whitespace/comments stripped
  • Full error recovery — always produces a tree, even from invalid input

4.2 Error Recovery Strategies

  • Synchronization: On error, skip tokens until a synchronization point (;, }, keyword)
  • Insertion: Insert expected tokens (e.g., missing closing brace)
  • Deletion: Skip unexpected tokens
  • Error nodes: Wrap invalid syntax in ErrorNode in the tree

5. Semantic Analysis (flux_sema)

5.1 Name Resolution

  • Build a scope tree from the AST
  • Resolve all identifiers to their declarations
  • Handle imports (use declarations)
  • Detect duplicate definitions and shadowing
  • Build the symbol table

5.2 Symbol Table

pub struct SymbolTable {
    symbols: HashMap<SymbolId, Symbol>,
    scopes: Vec<Scope>,
}

pub struct Symbol {
    pub id: SymbolId,
    pub name: String,
    pub kind: SymbolKind,
    pub visibility: Visibility,
    pub span: Span,
    pub type_id: Option<TypeId>,
}

pub enum SymbolKind {
    Variable, Function, Struct, Class, Enum, Trait, 
    TypeParam, Module, Constant, Field, Method, EnumVariant,
}

6. Type Checker (flux_types)

6.1 Algorithm

  • Extended Hindley-Milner with trait bounds
  • Bidirectional type inference (checking mode + synthesis mode)
  • Constraint generation → constraint solving → substitution application
  • Monomorphization tracking for generic instantiations

6.2 Key Operations

  • Synthesis: Compute the type of an expression bottom-up
  • Checking: Verify an expression has an expected type top-down
  • Unification: Find a substitution that makes two types equal
  • Trait resolution: Determine which impl block satisfies a trait bound
  • Exhaustiveness checking: Verify match expressions cover all cases

7. Mid-Level IR (flux_mir)

7.1 Design

MIR is a simplified, desugared representation of the program: - All control flow is explicit (no for loops — lowered to loop + iterator) - All pattern matching is lowered to decision trees - All method calls are resolved to concrete functions - All generics are monomorphized - Suitable for borrow checking and optimization

7.2 Borrow Checker

  • Track the liveness of each variable (when it is first used, last used)
  • Track all borrows (shared and mutable)
  • Verify borrowing rules at each program point
  • Detect use-after-move errors
  • Check lifetime constraints

8. Optimizer (flux_optim)

8.1 MIR-Level Optimizations (before LLVM)

  • Constant folding and propagation
  • Dead code elimination
  • Inlining (small functions)
  • Devirtualization (convert trait calls to direct calls when possible)
  • Copy elision
  • Unused allocation elimination

8.2 LLVM Optimization Passes

Flux delegates heavy optimization to LLVM:

Level Flag Passes
0 --opt-level 0 No optimization (fastest compilation)
1 --opt-level 1 Basic optimizations
2 --opt-level 2 Most optimizations (default for release)
3 --opt-level 3 Aggressive optimization (may increase binary size)
s --opt-level s Optimize for size
z --opt-level z Aggressively optimize for size

9. Code Generation (flux_codegen)

9.1 LLVM IR Generation

  • Translate optimized MIR to LLVM IR
  • Generate debug information (DWARF format)
  • Handle ABI differences across targets
  • Emit metadata for link-time optimization (LTO)

9.2 Target Support

pub struct TargetSpec {
    pub triple: String,        // e.g., "x86_64-unknown-linux-gnu"
    pub arch: Architecture,
    pub os: OperatingSystem,
    pub abi: Abi,
    pub pointer_width: u32,
    pub endian: Endian,
    pub features: Vec<String>, // CPU features (e.g., "avx2", "neon")
}

10. Diagnostics (flux_diagnostics)

10.1 Error Message Philosophy

  • Every error message must explain what went wrong
  • Every error message should suggest how to fix it
  • Use color and formatting to highlight relevant code
  • Show related code spans (e.g., where a variable was defined)

10.2 Diagnostic Structure

pub struct Diagnostic {
    pub severity: Severity,     // Error, Warning, Note, Help
    pub message: String,
    pub code: Option<String>,   // E0001, W0042, etc.
    pub primary_span: Span,
    pub labels: Vec<Label>,
    pub suggestions: Vec<Suggestion>,
    pub notes: Vec<String>,
}

11. Incremental Compilation

11.1 Strategy

  • File-level granularity: only re-analyze changed files
  • Dependency tracking: if a.flux imports from b.flux, changing b.flux triggers re-checking of a.flux
  • Content hashing: skip re-compilation if file content hash unchanged
  • Query-based architecture: computation results are cached by query key

11.2 Cache Structure

.flux-cache/
├── dep-graph.bin       # Dependency graph
├── queries/            # Cached query results
├── incremental/        # Incremental compilation data
│   ├── ast/            # Cached ASTs
│   ├── types/          # Cached type information
│   └── mir/            # Cached MIR
└── artifacts/          # Compiled object files

12. Parallel Compilation

12.1 Parallelism Points

  • File lexing/parsing: Each file can be lexed and parsed independently
  • Type checking: Independent modules can be type-checked in parallel
  • MIR lowering: Each function can be lowered independently
  • Code generation: Each function's LLVM IR can be generated independently
  • LLVM optimization: LLVM modules can be optimized in parallel

12.2 Thread Pool

Use a work-stealing thread pool (Rayon) for parallel compilation: - Default: number of available CPU cores - Configurable via FLUX_THREADS environment variable or Flux.toml
