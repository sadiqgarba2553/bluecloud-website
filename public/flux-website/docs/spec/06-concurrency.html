Flux Language Specification — Concurrency

Version: 0.1.0-draft
Status: Draft

1. Overview

Flux provides first-class concurrency support through async/await, structured concurrency, channels, and an actor model. All concurrency primitives are designed to be safe, composable, and efficient.

Design goals: - Prevent data races at compile time (via Send and Sync traits) - Make concurrent code look and feel like sequential code (via async/await) - Provide structured concurrency (no orphaned tasks) - Support both cooperative (async) and preemptive (thread) concurrency - Zero-cost abstractions for concurrency patterns

2. Async/Await

2.1 Async Functions

An async function returns a Future<T> that can be awaited:

async fn fetch_data(url: String) -> Result<String, HttpError> {
    let response = await http.get(url)?
    return Ok(response.body)
}

Async functions are lazy — calling an async function returns a Future but does not start execution. Execution begins only when the future is awaited.

let future = fetch_data("https://api.example.com")  // Not started yet
let result = await future                            // Starts and waits

2.2 Await Expressions

The await keyword suspends the current task until the future completes:

async fn process() {
    // Sequential awaits
    let a = await fetch("url1")
    let b = await fetch("url2")

    // Concurrent awaits (both futures start simultaneously)
    let (a, b) = await (fetch("url1"), fetch("url2"))

    // Await with timeout
    let result = await fetch("url").timeout(Duration.seconds(5))
}

2.3 Async Blocks

Anonymous async blocks create futures inline:

let future = async {
    let data = await fetch_data("url")
    return process(data)
}

let result = await future

3. Structured Concurrency

3.1 Task Groups

Task groups ensure all spawned tasks complete before the group exits. No orphaned tasks are possible.

async fn fetch_all_users(ids: List<Int>) -> List<User> {
    let group = TaskGroup<User>.new()

    for id in ids {
        group.spawn(async {
            return await fetch_user(id)
        })
    }

    // All tasks must complete before collect() returns
    return await group.collect()
}

3.2 Spawn

spawn creates a new concurrent task within the current scope:

async fn main() {
    // Spawn a background task
    let handle = spawn async {
        await long_running_operation()
        return 42
    }

    // Do other work...
    print("Working...")

    // Wait for the spawned task
    let result = await handle
    print("Result: {result}")
}

3.3 Task Cancellation

Tasks support cooperative cancellation:

async fn main() {
    let handle = spawn async {
        loop {
            if Task.is_cancelled() {
                print("Task cancelled")
                return
            }
            await do_work()
        }
    }

    await Task.sleep(Duration.seconds(5))
    handle.cancel()
    await handle
}

3.4 Scoped Tasks

Scoped tasks guarantee that all child tasks complete before the scope exits:

async fn process_batch(items: List<Item>) -> List<Result> {
    return await Task.scope(async |scope| {
        mut results = List<Future<Result>>.new()

        for item in items {
            let future = scope.spawn(async {
                return await process_item(item)
            })
            results.append(future)
        }

        return await Future.all(results)
    })
}

4. Channels

Channels provide message-passing communication between concurrent tasks.

4.1 Bounded Channels

async fn producer_consumer() {
    // Create a channel with buffer size 10
    let (tx, rx) = Channel<Int>.bounded(10)

    // Producer
    spawn async {
        for i in 0..100 {
            await tx.send(i)
        }
        tx.close()
    }

    // Consumer
    while let Some(value) = await rx.receive() {
        print("Got: {value}")
    }
    print("Channel closed")
}

4.2 Unbounded Channels

let (tx, rx) = Channel<String>.unbounded()

4.3 Multi-producer Multi-consumer

async fn fan_out_fan_in() {
    let (tx, rx) = Channel<WorkItem>.bounded(100)

    // Multiple producers
    for i in 0..4 {
        let tx = tx.clone()
        spawn async {
            for item in generate_work(i) {
                await tx.send(item)
            }
        }
    }

    // Multiple consumers
    for _ in 0..8 {
        let rx = rx.clone()
        spawn async {
            while let Some(item) = await rx.receive() {
                process(item)
            }
        }
    }
}

4.4 Select

Wait on multiple channels simultaneously:

async fn multiplexer() {
    let (tx1, rx1) = Channel<Int>.bounded(10)
    let (tx2, rx2) = Channel<String>.bounded(10)

    loop {
        match select {
            value from rx1 -> print("Int: {value}")
            message from rx2 -> print("String: {message}")
            timeout Duration.seconds(5) -> {
                print("Timeout!")
                break
            }
        }
    }
}

5. Thread Safety

5.1 Send and Sync Traits

Flux prevents data races at compile time through two marker traits:

Trait Meaning
Send Type can be safely transferred between threads
Sync Type can be safely shared between threads (via &T)

These traits are automatically implemented by the compiler when safe to do so. A type is Send if all its fields are Send. A type is Sync if all its fields are Sync.

// Rc<T> is NOT Send (not thread-safe reference counting)
let shared = Rc.new(42)
// spawn async { print(shared) }  // Compile error: Rc<Int> is not Send

// Arc<T> IS Send and Sync (atomic reference counting)
let shared = Arc.new(42)
spawn async { print(shared) }     // OK

5.2 Types That Are Not Send/Sync

Type Send Sync Reason
Rc<T> ❌ ❌ Non-atomic reference counting
Cell<T> ✅ ❌ Interior mutability without synchronization
RefCell<T> ✅ ❌ Runtime borrow checking, not thread-safe
*mut T ❌ ❌ Raw pointers have no safety guarantees
Arc<T> ✅ ✅ Atomic reference counting
Mutex<T> ✅ ✅ Synchronized access

6. Synchronization Primitives

6.1 Mutex

use flux.concurrent.Mutex

let counter = Arc.new(Mutex.new(0))

let handles = (0..10).map(fn(i) {
    let counter = counter.clone()
    return spawn async {
        mut lock = counter.lock()
        *lock += 1
    }
})

await Future.all(handles)
let final_count = counter.lock()
print("Count: {*final_count}")   // 10

6.2 RwLock

use flux.concurrent.RwLock

let data = Arc.new(RwLock.new(HashMap<String, Int>.new()))

// Multiple readers
spawn async {
    let read = data.read()
    print(read.get("key"))
}

// Exclusive writer
spawn async {
    let mut write = data.write()
    write.insert("key", 42)
}

6.3 Semaphore

use flux.concurrent.Semaphore

let sem = Arc.new(Semaphore.new(3))  // Allow 3 concurrent accesses

for i in 0..10 {
    let sem = sem.clone()
    spawn async {
        let permit = await sem.acquire()
        await process(i)
        // permit dropped — releases the semaphore
    }
}

6.4 Atomic Types

use flux.concurrent.{AtomicInt, AtomicBool, Ordering}

let counter = AtomicInt.new(0)
counter.fetch_add(1, Ordering.SeqCst)

let flag = AtomicBool.new(false)
flag.store(true, Ordering.Release)

if flag.load(Ordering.Acquire) {
    print("Flag is set")
}

6.5 Barrier

use flux.concurrent.Barrier

let barrier = Arc.new(Barrier.new(4))  // Wait for 4 tasks

for i in 0..4 {
    let barrier = barrier.clone()
    spawn async {
        // Phase 1
        do_work_phase1(i)

        // Wait for all tasks to finish phase 1
        await barrier.wait()

        // Phase 2 — all tasks proceed together
        do_work_phase2(i)
    }
}

7. Actor Model

Actors encapsulate state and communicate exclusively through messages:

actor Counter {
    mut count: Int = 0

    fn increment(mut self) {
        self.count += 1
    }

    fn decrement(mut self) {
        self.count -= 1
    }

    fn get_count(self) -> Int {
        return self.count
    }
}

async fn main() {
    let counter = Counter.new()

    // Messages are sent asynchronously
    await counter.increment()
    await counter.increment()
    await counter.increment()

    let count = await counter.get_count()
    print("Count: {count}")   // 3
}

Actor methods are processed sequentially — no data races are possible within an actor. Communication between actors happens via message passing.

8. Parallel Iterators

use flux.parallel

let results = items
    .par_iter()               // Parallel iterator
    .filter((item) => item.is_valid())
    .map((item) => expensive_transform(item))
    .collect()

// Control parallelism
let results = items
    .par_iter()
    .with_threads(4)
    .map(process)
    .collect()

9. Async Runtime

9.1 Executor

Flux includes a built-in async executor that manages task scheduling:

// The runtime is started automatically when main is async
async fn main() {
    await my_async_code()
}

// Manual runtime control (advanced)
fn main() {
    let runtime = Runtime.new(RuntimeConfig {
        worker_threads: 4
        max_blocking_threads: 512
    })

    runtime.block_on(async {
        await my_async_code()
    })
}

9.2 Timer and Sleep

async fn example() {
    // Sleep
    await Task.sleep(Duration.seconds(1))

    // Timeout
    let result = await fetch("url").timeout(Duration.seconds(5))

    // Interval
    let mut interval = Interval.new(Duration.milliseconds(100))
    for _ in 0..10 {
        await interval.tick()
        do_periodic_work()
    }
}

10. Memory Ordering

For low-level concurrent programming, Flux exposes memory ordering guarantees:

Ordering Guarantee
Relaxed No ordering guarantees (fastest)
Acquire Reads cannot be reordered after this
Release Writes cannot be reordered before this
AcqRel Both Acquire and Release
SeqCst Total sequential consistency (default, safest) 
use flux.concurrent.{AtomicBool, Ordering}

let ready = AtomicBool.new(false)
let data = Arc.new(Mutex.new(0))

// Writer thread
spawn async {
    let mut lock = data.lock()
    *lock = 42
    ready.store(true, Ordering.Release)
}

// Reader thread
spawn async {
    while not ready.load(Ordering.Acquire) {
        Task.yield()
    }
    let lock = data.lock()
    assert(*lock == 42)
}

11. Deadlock Prevention

Flux provides tools to prevent and detect deadlocks:

11.1 Lock Ordering

The linter can enforce a consistent lock ordering to prevent deadlocks:

#[lock_order(1)]
let lock_a = Mutex.new(0)

#[lock_order(2)]
let lock_b = Mutex.new(0)

// OK: acquiring in order
let a = lock_a.lock()
let b = lock_b.lock()

// Warning: acquiring in reverse order (potential deadlock)
// let b = lock_b.lock()
// let a = lock_a.lock()

11.2 Try-Lock

// Non-blocking lock attempt
match mutex.try_lock() {
    Some(guard) -> {
        // Got the lock
        *guard += 1
    }
    None -> {
        // Lock is held by someone else
        print("Could not acquire lock")
    }
}

11.3 Timeout Lock

match await mutex.lock_timeout(Duration.seconds(5)) {
    Some(guard) -> { /* got it */ }
    None -> { /* timed out */ }
}

12. Concurrency Patterns

12.1 Fan-Out/Fan-In

async fn fan_out_fan_in(items: List<Item>) -> List<Result> {
    let group = TaskGroup<Result>.new()
    for item in items {
        group.spawn(async { return await process(item) })
    }
    return await group.collect()
}

12.2 Pipeline

async fn pipeline(input: Channel.Receiver<Data>) -> Channel.Receiver<Output> {
    let (tx, rx) = Channel<Output>.bounded(100)

    spawn async {
        while let Some(data) = await input.receive() {
            let processed = await transform(data)
            await tx.send(processed)
        }
        tx.close()
    }

    return rx
}

12.3 Rate Limiting

async fn rate_limited_requests(urls: List<String>) {
    let limiter = RateLimiter.new(requests_per_second: 10)

    for url in urls {
        await limiter.wait()
        spawn async {
            await fetch(url)
        }
    }
}

12.4 Worker Pool

async fn worker_pool(items: List<WorkItem>, concurrency: Int) {
    let sem = Semaphore.new(concurrency)

    let group = TaskGroup<Void>.new()
    for item in items {
        let permit = await sem.acquire()
        group.spawn(async {
            defer permit.release()
            await process_item(item)
        })
    }

    await group.wait()
}
