Flux Language Specification — Introduction

Version: 0.1.0-draft
Status: Draft
Last Updated: 2026-06-29

1. Overview

Flux is a modern, general-purpose, statically typed, compiled programming language. It is designed to be approachable for beginners while powerful enough for systems programming, web development, artificial intelligence, game development, and everything in between.

Flux compiles to native machine code through LLVM, producing optimized binaries for all major platforms and architectures.

2. Design Philosophy

Flux is guided by seven core principles, listed in order of priority:

2.1 Clarity Over Cleverness

Code is read far more often than it is written. Flux prioritizes readable, self-documenting code over terse, clever expressions. There should be one obvious way to accomplish any task.

// Good — clear intent
let adults = users.filter(fn(u) { u.age >= 18 })

// Also good — named function for complex logic
fn is_eligible(user: User) -> Bool {
    return user.age >= 18 and user.is_verified
}
let eligible = users.filter(is_eligible)

2.2 Safety by Default

Flux prevents entire categories of bugs at compile time:

  • Null safety — none values must be explicitly handled via Option<T>
  • Thread safety — data races are prevented by the type system
  • Memory safety — ownership and borrowing prevent use-after-free, dangling pointers, and double-free errors
  • Immutability — variables are immutable by default; mutation requires the mut keyword
  • Exhaustive matching — match expressions must cover all cases

2.3 Performance Without Compromise

Flux produces optimized native binaries with:

  • Zero-cost abstractions — generics, traits, and iterators compile to the same code as hand-written equivalents
  • No garbage collector — hybrid ownership + ARC system with deterministic destruction
  • LLVM optimizations — leverages decades of LLVM optimization passes
  • Compile-time evaluation — comptime blocks execute at compile time

2.4 Progressive Disclosure

Simple programs should be simple. Advanced features should be available but never forced upon the programmer.

// A complete, valid Flux program — no boilerplate
fn main() {
    print("Hello, World!")
}

// When you need more power, it's there
async fn fetch_and_process<T: Deserialize>(url: String) -> Result<T, AppError> {
    let response = await http.get(url)?
    let data: T = json.parse(response.body)?
    return Ok(data)
}

2.5 Modern Tooling as a First-Class Citizen

The language includes official, integrated tooling:

  • Package manager — dependency resolution, publishing, private registries
  • Formatter — canonical code formatting (no style debates)
  • Linter — static analysis with auto-fix suggestions
  • Test runner — unit, integration, and benchmark testing
  • Documentation generator — generate API docs from code comments
  • Language server — IDE support via LSP

2.6 Universal Platform Support

One language for every platform and every domain:

  • Desktop applications (Windows, macOS, Linux)
  • Mobile applications (Android, iOS)
  • Web servers and APIs
  • WebAssembly for browser-side code
  • Embedded systems and IoT
  • Game development
  • AI/ML workloads
  • Operating system development

2.7 Interoperability

Flux is designed to coexist with existing ecosystems:

  • Seamless C FFI for leveraging existing libraries
  • C++ interop for performance-critical codebases
  • Python bindings for AI/ML ecosystems
  • WebAssembly compilation for web integration
  • Platform-native APIs (Win32, Cocoa, GTK, Android NDK, UIKit)

3.2 Tier Definitions

  • Tier 1 — Fully supported and tested in CI. Guaranteed to compile and produce correct binaries. Official pre-built binaries available.
  • Tier 2 — Supported and expected to work. Tested periodically. Community builds may be available.
  • Tier 3 — Experimental. May compile. Community maintained.

6. Notation Conventions

Throughout this specification, the following conventions are used:

6.1 Grammar Notation

The formal grammar uses Parsing Expression Grammar (PEG) notation:

Notation Meaning
'literal' Literal string match
[a-z] Character class
A B Sequence (A followed by B)
A / B Ordered choice (try A, then B)
A* Zero or more repetitions
A+ One or more repetitions
A? Optional (zero or one)
!A Negative lookahead
&A Positive lookahead
(A) Grouping

6.2 Code Examples

All code examples in this specification are valid Flux programs or fragments. Examples are formatted as:

// This is a Flux code example
fn example() {
    let x = 42
    print(x)
}

6.3 Type Signatures

Type signatures are written in the form:

fn name(param: Type, ...) -> ReturnType

6.4 Terminology

Term Definition
Expression A construct that evaluates to a value
Statement A construct that performs an action but does not produce a value
Declaration A statement that introduces a new name binding
Block A sequence of statements enclosed in { }
Item A top-level declaration (function, struct, enum, trait, etc.)
Crate A compilation unit (equivalent to a package)
Module A namespace within a crate

7. Document Organization

The Flux Language Specification is organized into the following sections:

  1. Introduction (this document) — Philosophy, goals, overview
  2. Lexical Structure — Tokens, keywords, literals, operators
  3. Syntax Grammar — Complete formal grammar, all constructs
  4. Type System — Types, generics, inference, subtyping
  5. Memory Model — Ownership, borrowing, ARC, lifetimes
  6. Concurrency — Async/await, tasks, channels, thread safety
  7. Modules and Packages — Import system, visibility, package manifest
  8. Error Handling — Result, Option, error propagation
  9. Metaprogramming — Macros, comptime, reflection, attributes
  10. Interoperability — C FFI, platform bindings, WebAssembly

Each section is self-contained but cross-references related sections where appropriate.

8. Versioning

This specification follows the language version. The current version is 0.1.0-draft.

Language changes follow the RFC process documented in CONTRIBUTING.md. All breaking changes require a major version bump after 1.0.

The specification uses the following stability markers:

Marker Meaning
Stable Finalized, will not change without a major version bump
Unstable May change in future versions
Draft Under active development, subject to significant changes
Deprecated Will be removed in a future version
Experimental Available behind a feature flag, not recommended for production
