Flux Language Specification — Error Handling

Version: 0.1.0-draft
Status: Draft

1. Overview

Flux uses errors as values as its primary error handling strategy. Errors are represented by the Result<T, E> and Option<T> types, making error paths explicit and impossible to ignore.

Design principles: - Errors are not exceptional — they are expected and handled as values - The type system enforces error handling (you cannot accidentally ignore an error) - Error propagation is ergonomic with the ? operator - try/catch provides familiar syntax as sugar over pattern matching - Panics exist for truly unrecoverable situations only

2. Result\<T, E>

2.1 Definition

enum Result<T, E> {
    Ok(T),
    Err(E),
}

Result is the standard return type for operations that can fail:

fn parse_int(s: String) -> Result<Int, ParseError> {
    // ... parsing logic ...
    if valid {
        return Ok(value)
    }
    return Err(ParseError.InvalidFormat(s))
}

2.2 Working with Result

let result = parse_int("42")

// Pattern matching
match result {
    Ok(value) -> print("Parsed: {value}")
    Err(error) -> print("Error: {error}")
}

// Methods
let value = result.unwrap()               // Panics on Err
let value = result.unwrap_or(0)           // Default on Err
let value = result.unwrap_or_else(fn(e) { log(e); 0 })
let value = result.expect("must parse")   // Panics with message on Err
let mapped = result.map(fn(v) { v * 2 })  // Transform Ok value
let mapped = result.map_err(fn(e) { AppError.from(e) })  // Transform Err
let flat = result.and_then(fn(v) { validate(v) })  // Chain operations
let is_ok = result.is_ok()
let is_err = result.is_err()

// If-let
if let Ok(value) = parse_int("42") {
    print("Got: {value}")
}

3. Option\<T>

3.1 Definition

enum Option<T> {
    Some(T),
    None,
}

Option represents a value that may or may not exist. It replaces null/nil entirely.

3.2 Working with Option

fn find(items: List<Int>, target: Int) -> Option<Int> {
    for (index, item) in items.enumerate() {
        if item == target {
            return Some(index)
        }
    }
    return None
}

let index = find([10, 20, 30], 20)

// Pattern matching
match index {
    Some(i) -> print("Found at index {i}")
    None    -> print("Not found")
}

// Shorthand type syntax
let name: String? = get_name()   // String? == Option<String>

// Methods
let value = name.unwrap()
let value = name.unwrap_or("Anonymous")
let value = name ?? "Anonymous"            // None-coalescing operator
let upper = name.map(fn(n) { n.to_upper() })
let flat = name.and_then(fn(n) { validate_name(n) })

// Optional chaining
let city = user?.address?.city             // Returns Option<String>
let length = user?.name?.length ?? 0

// If-let
if let Some(n) = name {
    print("Name: {n}")
}

// While-let
while let Some(item) = iterator.next() {
    process(item)
}

4. Error Propagation (?)

The ? operator provides ergonomic error propagation. When applied to a Result, it: - Unwraps Ok(value) and continues execution - Returns Err(error) early from the current function (with automatic error conversion)

fn read_config(path: String) -> Result<Config, AppError> {
    let content = File.read(path)?          // Returns Err if file read fails
    let parsed = json.parse(content)?       // Returns Err if parsing fails
    let config = Config.from(parsed)?       // Returns Err if conversion fails
    return Ok(config)
}

// Without `?` — equivalent but verbose:
fn read_config_verbose(path: String) -> Result<Config, AppError> {
    let content = match File.read(path) {
        Ok(c) -> c
        Err(e) -> return Err(AppError.from(e))
    }
    let parsed = match json.parse(content) {
        Ok(p) -> p
        Err(e) -> return Err(AppError.from(e))
    }
    let config = match Config.from(parsed) {
        Ok(c) -> c
        Err(e) -> return Err(AppError.from(e))
    }
    return Ok(config)
}

4.1 ? on Option

The ? operator also works on Option, returning None early:

fn get_user_city(users: HashMap<Int, User>, id: Int) -> Option<String> {
    let user = users.get(id)?          // Returns None if user not found
    let address = user.address?        // Returns None if no address
    return Some(address.city)
}

4.2 Automatic Error Conversion

The ? operator automatically converts errors using the From trait:

enum AppError {
    Io(IoError),
    Parse(ParseError),
    Validation(String),
}

impl From<IoError> for AppError {
    fn from(e: IoError) -> AppError = AppError.Io(e)
}

impl From<ParseError> for AppError {
    fn from(e: ParseError) -> AppError = AppError.Parse(e)
}

// Now `?` automatically converts IoError and ParseError to AppError
fn process() -> Result<Data, AppError> {
    let content = File.read("data.txt")?     // IoError → AppError
    let parsed = json.parse(content)?        // ParseError → AppError
    return Ok(parsed)
}

5. Try-Catch

try/catch provides familiar exception-style syntax as sugar over Result matching:

try {
    let file = File.open("data.txt")?
    let content = file.read_all()?
    let data = json.parse(content)?
    process(data)
} catch IoError.NotFound {
    print("File not found — using defaults")
    use_defaults()
} catch IoError.PermissionDenied {
    print("Permission denied")
} catch ParseError as e {
    print("Parse error: {e.message}")
} catch error {
    print("Unexpected error: {error}")
}

5.1 Try-Catch with Value

let config = try {
    load_config("config.toml")?
} catch {
    Config.default()   // Fallback value
}

5.2 Desugaring

The try/catch block desugars to a match expression:

// This try/catch:
try {
    let x = operation()?
    use(x)
} catch SomeError as e {
    handle(e)
}

// Desugars to:
match (fn() -> Result<Void, Error> {
    let x = operation()?
    use(x)
    return Ok(())
})() {
    Ok(()) -> {}
    Err(e) if e is SomeError -> handle(e as SomeError)
    Err(e) -> panic("Unhandled error: {e}")
}

6. Custom Error Types

6.1 Error Enums

enum DatabaseError {
    ConnectionFailed(host: String, port: Int),
    QueryFailed(query: String, reason: String),
    NotFound(table: String, id: Int),
    Timeout(duration: Duration),
}

impl Display for DatabaseError {
    fn to_string(self) -> String {
        match self {
            ConnectionFailed(host, port) -> "Failed to connect to {host}:{port}"
            QueryFailed(query, reason)   -> "Query failed: {reason}\nQuery: {query}"
            NotFound(table, id)          -> "Record not found: {table}#{id}"
            Timeout(duration)            -> "Operation timed out after {duration}"
        }
    }
}

6.2 Error Trait

trait Error: Display {
    fn source(self) -> Option<&Error> {
        return None
    }
}

impl Error for DatabaseError {}

6.3 Error Chaining

enum AppError {
    Database(DatabaseError),
    Config(ConfigError),
    Internal(String),
}

impl Error for AppError {
    fn source(self) -> Option<&Error> {
        match self {
            AppError.Database(e) -> Some(&e)
            AppError.Config(e)   -> Some(&e)
            AppError.Internal(_) -> None
        }
    }
}

// Adding context to errors
fn load_user(id: Int) -> Result<User, AppError> {
    let data = db.query("SELECT * FROM users WHERE id = {id}")
        .map_err(fn(e) { AppError.Database(e) })?

    return Ok(User.from(data))
}

6.4 The context Extension

// Add human-readable context to any error
fn setup() -> Result<Void, AppError> {
    File.read("config.toml")
        .context("Failed to read configuration file")?

    db.connect(config.db_url)
        .context("Failed to connect to database")?

    return Ok(())
}

7. Panic

7.1 When to Panic

Panics are for unrecoverable errors — situations that indicate a programming bug, not an expected failure:

fn get(self, index: Int) -> T {
    if index < 0 or index >= self.length {
        panic("Index out of bounds: {index} (length: {self.length})")
    }
    return self.data[index]
}

7.2 Panic Behavior

  1. Prints error message with stack trace to stderr
  2. Unwinds the stack, running Drop implementations
  3. Exits the process with non-zero exit code
thread 'main' panicked at 'Index out of bounds: 5 (length: 3)'
  at src/collections/list.flux:42:9

Stack trace:
  0: flux.collections.List<Int>.get (src/collections/list.flux:42)
  1: main (src/main.flux:15)

7.3 Catching Panics

In rare cases, panics can be caught (e.g., in test runners or server frameworks):

let result = panic.catch(fn() {
    dangerous_operation()
})

match result {
    Ok(value)  -> print("Success: {value}")
    Err(panic) -> print("Caught panic: {panic.message}")
}

7.4 Assert

assert(x > 0)                           // Panics if false
assert(x > 0, "x must be positive")     // With message
debug_assert(expensive_check())          // Only in debug builds

8. Error Handling Patterns

8.1 The "?" Chain Pattern

fn process_request(req: Request) -> Result<Response, AppError> {
    let user = authenticate(req)?
    let data = fetch_data(user)?
    let result = transform(data)?
    return Ok(Response.json(result))
}

8.2 The "Match and Handle" Pattern

fn resilient_operation() -> Data {
    match try_primary() {
        Ok(data) -> return data
        Err(e) -> {
            log.warn("Primary failed: {e}, trying fallback")
            match try_fallback() {
                Ok(data) -> return data
                Err(e) -> {
                    log.error("Fallback also failed: {e}")
                    return Data.default()
                }
            }
        }
    }
}

8.3 The "Collect Results" Pattern

fn process_all(items: List<Item>) -> Result<List<Output>, AppError> {
    // Collect all results, failing on the first error
    let results: Result<List<Output>, AppError> = items
        .map(fn(item) { process(item) })
        .collect()

    return results
}

// Or collect errors separately:
fn process_all_lenient(items: List<Item>) -> (List<Output>, List<AppError>) {
    mut successes = List<Output>.new()
    mut errors = List<AppError>.new()

    for item in items {
        match process(item) {
            Ok(output) -> successes.append(output)
            Err(error) -> errors.append(error)
        }
    }

    return (successes, errors)
}
