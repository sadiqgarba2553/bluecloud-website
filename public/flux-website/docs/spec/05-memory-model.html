Flux Language Specification — Memory Model

Version: 0.1.0-draft
Status: Draft

1. Overview

Flux uses a hybrid memory management system that combines the strengths of Rust's ownership model with Swift's Automatic Reference Counting (ARC). The goal is maximum safety and performance without a traditional garbage collector.

Design principles: - No segmentation faults - No dangling pointers - No double-free errors - No data races - Deterministic resource cleanup - Minimal runtime overhead - Predictable performance (no GC pauses)

2. Stack vs. Heap Allocation

2.1 Stack Allocation (Default)

Value types are allocated on the stack by default:

Type Allocation
Primitives (Int, Float64, Bool, Char) Stack
Tuples Stack
Fixed-size arrays [T; N] Stack
Structs (by default) Stack
Enums (without heap data) Stack
References (&T, &mut T) Stack (pointer itself) 
fn example() {
    let x = 42              // Stack: 8 bytes
    let point = Point { x: 1.0, y: 2.0 }  // Stack: 16 bytes
    let pair = (10, 20)     // Stack: 16 bytes
}   // All deallocated when function returns — zero cost

2.2 Heap Allocation

Reference types and dynamically-sized data live on the heap:

Type Allocation
Classes Heap (managed by ARC)
String Heap (small-string optimization for ≤23 bytes)
List<T> Heap (growable buffer)
HashMap<K, V> Heap
Box<T> Heap (explicit boxing)
Rc<T> Heap (reference counted)
Arc<T> Heap (atomic reference counted) 
fn example() {
    let name = "Hello, World!"    // Heap: String buffer
    let items = [1, 2, 3]         // Heap: List buffer
    let dog = Dog(name: "Rex")    // Heap: class instance
}   // ARC decrements reference counts; memory freed when count reaches 0

3. Ownership

3.1 Ownership Rules

Every value in Flux has exactly one owner at any time. Ownership is the fundamental mechanism for memory safety.

Rules: 1. Each value has exactly one owner 2. When the owner goes out of scope, the value is dropped (freed) 3. Ownership can be transferred (moved) to another variable 4. After a move, the original variable is no longer valid

3.2 Move Semantics

By default, assigning a value or passing it to a function moves ownership:

let name = "Alice"       // `name` owns the String
let greeting = name      // Ownership moves to `greeting`
// print(name)           // Compile error: `name` has been moved

fn take_ownership(s: String) {
    print(s)
}   // `s` is dropped here

let message = "Hello"
take_ownership(message)  // Ownership moves to `s`
// print(message)        // Compile error: `message` has been moved

3.3 Copy Types

Types that implement the Copy trait are copied instead of moved. All primitive types are Copy:

let x = 42
let y = x       // Copy — both x and y are valid
print(x)        // OK: Int is Copy
print(y)        // OK

// Copy types:
// Int, Int8, Int16, Int32, Int64
// UInt8, UInt16, UInt32, UInt64
// Float32, Float64
// Bool, Char, Byte
// Tuples of Copy types
// Fixed-size arrays of Copy types

3.4 Clone

For non-Copy types, explicit cloning creates a deep copy:

let a = "Hello"
let b = a.clone()    // Deep copy — both valid
print(a)             // OK
print(b)             // OK

// Structs with #[derive(Clone)]
#[derive(Clone)]
struct Config {
    name: String
    values: List<Int>
}

let config1 = Config { name: "dev", values: [1, 2, 3] }
let config2 = config1.clone()

4. Borrowing and References

4.1 Shared References (&T)

A shared reference borrows a value without taking ownership. Multiple shared references can exist simultaneously.

fn print_length(s: &String) {
    print("Length: {s.length}")
}   // `s` is a reference, so the String is NOT dropped

let name = "Alice"
print_length(&name)    // Borrow `name`
print(name)            // OK: `name` is still valid

4.2 Mutable References (&mut T)

A mutable reference allows modification of the borrowed value. Only one mutable reference can exist at a time, and no shared references can coexist with it.

fn append_exclamation(s: &mut String) {
    s.push('!')
}

mut greeting = "Hello"
append_exclamation(&mut greeting)
print(greeting)   // "Hello!"

4.3 Borrowing Rules

The compiler enforces these rules at compile time:

  1. At any given time, you can have either:
  2. Any number of shared references (&T), OR

  3. Exactly one mutable reference (&mut T) — but not both simultaneously.

  4. References must always be valid:

  5. A reference cannot outlive the data it refers to (no dangling references)
  6. The compiler tracks lifetimes to enforce this
mut data = [1, 2, 3]

let r1 = &data       // OK: shared reference
let r2 = &data       // OK: multiple shared references
print(r1)
print(r2)

let r3 = &mut data   // OK: r1 and r2 are no longer used
r3.append(4)

// let r4 = &data     // ERROR if r3 is still in use
// Cannot have shared and mutable references simultaneously

4.4 Method Receivers

Methods declare how they borrow self:

Receiver Meaning
self Takes ownership (consumes the value)
&self Borrows immutably (equivalent to self: &Self)
mut self Takes ownership with interior mutability
&mut self Borrows mutably 
impl Stack<T> {
    fn is_empty(&self) -> Bool {        // Borrows immutably
        return self.items.length == 0
    }

    fn push(&mut self, item: T) {       // Borrows mutably
        self.items.append(item)
    }

    fn into_list(self) -> List<T> {     // Consumes the stack
        return self.items
    }
}

5. Lifetimes

5.1 Lifetime Concept

A lifetime is the scope during which a reference is valid. The compiler infers lifetimes automatically in most cases.

fn longest(a: &String, b: &String) -> &String {
    if a.length > b.length {
        return a
    }
    return b
}

5.2 Lifetime Annotations

When the compiler cannot infer lifetimes, explicit annotations are needed:

fn longest<'a>(a: &'a String, b: &'a String) -> &'a String {
    if a.length > b.length {
        return a
    }
    return b
}

5.3 Lifetime Elision Rules

Flux has more aggressive lifetime elision than Rust, handling most cases automatically:

  1. Each input reference parameter gets its own lifetime
  2. If there is exactly one input lifetime, it is assigned to all output references
  3. If there is a &self or &mut self parameter, its lifetime is assigned to all output references
// These are equivalent — lifetime elision makes annotations unnecessary:

fn first_word(s: &String) -> &String { ... }
fn first_word<'a>(s: &'a String) -> &'a String { ... }

// Method with &self — lifetime is inferred:
impl Config {
    fn name(&self) -> &String {
        return &self.name
    }
}

5.4 Static Lifetime

The 'static lifetime indicates data that lives for the entire program execution:

const GREETING: &String = "Hello, World!"   // 'static lifetime

static CONFIG: Config = Config { ... }       // 'static lifetime

6. Automatic Reference Counting (ARC)

6.1 When ARC Is Used

ARC is used for heap-allocated reference types:

  • Classes — all class instances are ARC-managed
  • Rc<T> — explicit single-threaded reference counting
  • Arc<T> — explicit thread-safe reference counting

ARC is not used for: - Stack-allocated values (primitives, structs, tuples) - Values with a single owner - Borrowed references

6.2 How ARC Works

class Document {
    title: String
    content: String
}

fn example() {
    let doc1 = Document(title: "Hello", content: "World")
    // Reference count: 1

    let doc2 = doc1
    // Reference count: 2 (doc1 and doc2 point to the same object)

    // doc2 goes out of scope → reference count: 1
    // doc1 goes out of scope → reference count: 0 → memory freed
}

6.3 Weak References

To break reference cycles, use Weak<T>:

class Node {
    value: Int
    children: List<Node>
    parent: Weak<Node>      // Weak reference — does not increment count

    init(value: Int) {
        self.value = value
        self.children = List.new()
        self.parent = Weak.none()
    }
}

fn example() {
    let parent = Node(value: 1)
    let child = Node(value: 2)

    parent.children.append(child)
    child.parent = Weak.from(parent)

    // No reference cycle: child.parent is weak
}

6.4 Cycle Detection

Flux includes an optional compile-time cycle detection analysis that warns about potential reference cycles:

class A {
    b: B?       // Warning: potential cycle — A -> B -> A
}

class B {
    a: A?       // Warning suggests: make this Weak<A>
}

The compiler emits a warning when two class types hold strong references to each other. The fix is to make one reference Weak<T>.

7. Smart Pointers

7.1 Box\<T>

Box<T> allocates a value on the heap with single ownership:

let boxed = Box.new(42)         // Heap-allocated Int
let value = *boxed              // Dereference

// Primary use: recursive types
enum List<T> {
    Cons(T, Box<List<T>>),
    Nil,
}

7.2 Rc\<T> (Reference Counted)

Single-threaded shared ownership:

let shared = Rc.new("shared data")
let clone1 = shared.clone()     // Increments reference count
let clone2 = shared.clone()     // Increments reference count

print(Rc.strong_count(&shared))  // 3

7.3 Arc\<T> (Atomic Reference Counted)

Thread-safe shared ownership:

let shared = Arc.new(Config { ... })

spawn async {
    let config = shared.clone()
    // Safe to use across threads
    print(config.name)
}

7.4 Weak\<T>

Non-owning reference that doesn't prevent deallocation:

let strong = Rc.new(42)
let weak = Rc.downgrade(&strong)

// Upgrade to a strong reference (returns Option)
if let Some(value) = weak.upgrade() {
    print("Value: {value}")
} else {
    print("Value has been deallocated")
}

8. RAII and Deterministic Destruction

8.1 Drop Trait

Types can implement Drop for custom cleanup:

trait Drop {
    fn drop(mut self)
}

class FileHandle {
    fd: Int

    init(path: String) {
        self.fd = sys_open(path)
    }

    impl Drop {
        fn drop(mut self) {
            sys_close(self.fd)
            print("File closed")
        }
    }
}

fn example() {
    let file = FileHandle("data.txt")
    // ... use file ...
}   // file.drop() called automatically here

8.2 Drop Order

  • Local variables are dropped in reverse declaration order (LIFO)
  • Struct fields are dropped in declaration order
  • Array elements are dropped in forward order
fn example() {
    let a = Resource("A")    // Dropped third
    let b = Resource("B")    // Dropped second
    let c = Resource("C")    // Dropped first
}
// Drop order: C, B, A

8.3 Defer

defer provides explicit control over cleanup order:

fn database_operation() {
    let conn = db.connect()
    defer conn.close()          // Guaranteed to run on scope exit

    let tx = conn.begin()
    defer tx.rollback()         // Runs before conn.close()

    // ... operations ...

    tx.commit()                 // If this succeeds, rollback is a no-op
}

9. Unsafe Code

9.1 Unsafe Blocks

unsafe blocks allow operations that the compiler cannot verify as safe:

unsafe {
    let ptr = malloc(1024) as *mut Byte
    ptr.write(0, 42)
    let value = ptr.read(0)
    free(ptr as *mut Byte)
}

9.2 Unsafe Operations

The following operations require an unsafe block:

Operation Reason
Dereferencing a raw pointer May be null, dangling, or unaligned
Calling an unsafe fn Function has safety preconditions
Accessing a static mut May cause data races
Calling FFI (extern) functions No safety guarantees from external code
Transmuting types Can violate type safety
Inline assembly Unrestricted hardware access

9.3 Raw Pointers

let x = 42
let ptr: *Int = &x as *Int         // Shared raw pointer
let mut_ptr: *mut Int = &mut x as *mut Int  // Mutable raw pointer

unsafe {
    print(*ptr)          // Dereference
    *mut_ptr = 100       // Write through pointer
}

9.4 Unsafe Functions

unsafe fn dangerous_operation(ptr: *mut Byte, len: USize) {
    // Caller must guarantee:
    // - ptr is non-null
    // - ptr points to at least `len` bytes of valid memory
    // - No other references to this memory exist
    for i in 0..len {
        ptr.offset(i).write(0)
    }
}

10. Memory Safety Guarantees

10.1 What Flux Prevents (in safe code)

Bug Class Prevention Mechanism
Null pointer dereference Option<T> — no null references
Use after free Ownership — compiler tracks lifetimes
Double free Ownership — each value has exactly one owner
Dangling pointer Borrowing — references cannot outlive data
Buffer overflow Bounds checking on array/slice access
Data race Send/Sync — enforced at compile time
Uninitialized memory All variables must be initialized
Memory leak ARC + ownership — mostly prevented (cycles are warned about)

10.2 What Requires unsafe

Scenario Why It's Unsafe
FFI calls External code has no Flux safety guarantees
Raw pointer operations No compiler-verified bounds or validity
Global mutable state Potential data races
Type transmutation Can break type invariants
Inline assembly Direct hardware manipulation

10.3 Unsafe Does Not Mean Dangerous

unsafe means "I, the programmer, am taking responsibility for these safety invariants." Well-written unsafe code should:

  1. Be as small as possible
  2. Be encapsulated behind a safe API
  3. Be thoroughly documented with safety comments
  4. Be extensively tested
// Good: unsafe implementation behind a safe API
pub struct SafeBuffer {
    ptr: *mut Byte
    len: USize
    cap: USize
}

impl SafeBuffer {
    pub fn new(capacity: USize) -> SafeBuffer {
        let ptr = unsafe { malloc(capacity) as *mut Byte }
        return SafeBuffer { ptr: ptr, len: 0, cap: capacity }
    }

    pub fn push(&mut self, byte: Byte) {
        if self.len >= self.cap {
            self.grow()
        }
        // SAFETY: we just ensured len < cap, so this write is in bounds
        unsafe {
            self.ptr.offset(self.len).write(byte)
        }
        self.len += 1
    }

    impl Drop {
        fn drop(mut self) {
            unsafe { free(self.ptr as *mut Byte) }
        }
    }
}
