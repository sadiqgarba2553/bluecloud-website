Flux Language Specification — Lexical Structure

Version: 0.1.0-draft
Status: Draft

1. Source Encoding

All Flux source files are encoded in UTF-8. The compiler rejects files with invalid UTF-8 sequences.

Source files use the .flux extension.

The compiler recognizes the following line endings and normalizes them internally to \n: - \n (Unix/macOS) - \r\n (Windows) - \r (legacy macOS)

2. Lexical Grammar Overview

A Flux source file is a sequence of tokens. The lexer (tokenizer) converts raw source text into a stream of tokens. Whitespace and comments are discarded during tokenization (except for their role in separating tokens).

source_file = token*
token       = keyword / identifier / literal / operator / delimiter / comment

3. Whitespace

Whitespace characters are used to separate tokens and are otherwise ignored. Flux is not whitespace-sensitive (unlike Python).

whitespace = [ \t\r\n]+
Character Name Unicode
Space U+0020
\t Tab U+0009
\r Carriage Return U+000D
\n Line Feed U+000A

Flux uses curly braces for block structure, not indentation. However, the official formatter (flux format) enforces consistent indentation of 4 spaces (no tabs).

4. Comments

Flux supports three styles of comments:

4.1 Line Comments

// This is a line comment
let x = 42  // Inline comment

Grammar:

line_comment = '//' (!'\n' .)*

4.2 Block Comments

Block comments can span multiple lines and can be nested.

/* This is a block comment */

/*
 * Multi-line block comment.
 * The asterisks on each line are conventional but not required.
 */

/* Nested /* comments */ are supported */

Grammar:

block_comment = '/*' (block_comment / !'*/' .)* '*/'

4.3 Documentation Comments

Documentation comments are attached to the following item and are used by the documentation generator.

/// Computes the factorial of a non-negative integer.
///
/// # Parameters
/// - `n` — The integer to compute the factorial of.
///
/// # Returns
/// The factorial of `n`.
///
/// # Examples
/// ```flux
/// assert(factorial(5) == 120)
/// assert(factorial(0) == 1)
/// ```
fn factorial(n: Int) -> Int {
    if n <= 1 {
        return 1
    }
    return n * factorial(n - 1)
}

Module-level documentation uses //!:

//! This module provides utilities for working with dates and times.
//! It supports parsing, formatting, and arithmetic operations on
//! temporal values.

use flux.time.Duration

Grammar:

doc_comment    = '///' (!'\n' .)*
module_comment = '//!' (!'\n' .)*

6. Identifiers

6.1 Rules

Identifiers are names for variables, functions, types, modules, and other program entities.

identifier      = ident_start ident_continue*
ident_start     = [a-zA-Z_] / unicode_letter
ident_continue  = [a-zA-Z0-9_] / unicode_letter / unicode_digit
  • Identifiers are case-sensitive: foo, Foo, and FOO are distinct
  • Identifiers must not be keywords (see §5)
  • Identifiers beginning with _ are considered unused (suppress unused-variable warnings)
  • The single identifier _ is the wildcard pattern (discards a value)

6.2 Unicode Support

Identifiers may contain Unicode letters and digits, enabling code in non-English languages:

let π = 3.14159
let 名前 = "Flux"
let café = "☕"

The definition of "Unicode letter" and "Unicode digit" follows Unicode Standard categories: - Letter: Lu, Ll, Lt, Lm, Lo, Nl - Digit: Nd

6.3 Naming Conventions

While not enforced by the compiler (only by the linter), Flux follows these conventions:

Entity Convention Example
Variables snake_case user_count
Functions snake_case get_user
Types (struct, class, enum, trait) PascalCase HttpClient
Constants SCREAMING_SNAKE_CASE MAX_RETRIES
Modules snake_case flux.http
Type parameters Single uppercase or PascalCase T, Key, Value
Lifetimes 'lowercase 'a, 'src

7. Literals

7.1 Integer Literals

integer_literal = decimal_literal / hex_literal / octal_literal / binary_literal
decimal_literal = [0-9] [0-9_]*
hex_literal     = '0x' [0-9a-fA-F] [0-9a-fA-F_]*
octal_literal   = '0o' [0-7] [0-7_]*
binary_literal  = '0b' [01] [01_]*

Underscores may be used as visual separators in numeric literals. They are ignored by the compiler.

let million = 1_000_000
let flags = 0b1010_0011
let color = 0xFF_AA_00
let permissions = 0o755

Integer literals have no fixed type until inference determines one. The default type is Int (platform-native signed integer, at least 64 bits).

An optional type suffix can force a specific integer type:

let a = 42          // Int (inferred)
let b = 42i8        // Int8
let c = 42u32       // UInt32
let d = 42i64       // Int64

7.2 Floating-Point Literals

float_literal   = decimal_literal '.' decimal_literal exponent?
                / decimal_literal exponent
exponent        = [eE] [+-]? decimal_literal

let pi = 3.14159
let avogadro = 6.022e23
let tiny = 1.0e-10
let formatted = 1_234.567_89

The default floating-point type is Float64. Type suffixes:

let a = 3.14        // Float64 (inferred)
let b = 3.14f32     // Float32
let c = 3.14f64     // Float64

7.3 Boolean Literals

bool_literal = 'true' / 'false'

let enabled = true
let visible = false

7.4 Character Literals

Character literals represent a single Unicode scalar value, enclosed in single quotes:

char_literal = "'" (escape_sequence / !["'\\"] .) "'"

let letter = 'A'
let emoji = '🔥'
let newline = '\n'
let unicode = '\u{1F600}'

7.5 String Literals

Regular Strings

String literals are enclosed in double quotes and support escape sequences:

string_literal = '"' string_content* '"'
string_content = escape_sequence / interpolation / !["\\\n] .

let greeting = "Hello, World!"
let path = "C:\\Users\\flux"
let multiline = "Line 1\nLine 2"

Escape Sequences

Sequence Meaning
\\ Backslash
\" Double quote
\' Single quote
\n Newline (LF)
\r Carriage return (CR)
\t Tab
\0 Null character
\u{XXXX} Unicode scalar (1–6 hex digits)

String Interpolation

Expressions can be embedded in strings using curly braces:

interpolation = '{' expression '}'

let name = "Flux"
let version = 1
print("Welcome to {name} version {version}!")
print("2 + 2 = {2 + 2}")
print("User: {user.name}, Age: {user.age}")

To include a literal { or } in a string, double them:

let json = "{{\"key\": \"value\"}}"  // {"key": "value"}

Raw Strings

Raw strings do not process escape sequences. They are prefixed with r:

let regex = r"^\d{3}-\d{4}$"
let path = r"C:\Users\flux\file.txt"

Raw strings do not support interpolation. To combine raw and interpolation, use r strings with explicit concatenation.

Multi-line Strings

Multi-line strings use triple quotes and automatically strip common leading whitespace:

let html = """
    <html>
        <body>
            <h1>Hello</h1>
        </body>
    </html>
    """
// Result has no leading indentation; the indentation is relative to the closing """

Multi-line strings support interpolation:

let page = """
    <html>
        <body>
            <h1>{title}</h1>
            <p>{content}</p>
        </body>
    </html>
    """

7.6 None Literal

none_literal = 'none'

none represents the absence of a value and is the single inhabitant of the None type. It is used with Option<T>:

let result: Option<Int> = none

7.7 Collection Literals

// Array literal
let numbers = [1, 2, 3, 4, 5]

// Map literal
let colors = {
    "red": "#FF0000"
    "green": "#00FF00"
    "blue": "#0000FF"
}

// Set literal
let unique = Set{1, 2, 3, 4, 5}

// Tuple literal
let point = (10, 20)
let named = (x: 10, y: 20)

9.1 Trailing Commas

Trailing commas are allowed (and encouraged by the formatter) in all comma-separated lists:

let colors = [
    "red",
    "green",
    "blue",   // trailing comma is valid
]

fn create_user(
    name: String,
    email: String,
    age: Int,   // trailing comma is valid
) -> User {
    // ...
}

9.2 Comma Elision

In struct literals, map literals, and multi-line function parameters, commas may be omitted when each entry is on its own line:

// Commas optional — each field on its own line
let user = User {
    name: "Alice"
    email: "alice@example.com"
    age: 30
}

// Commas required — same line
let user = User { name: "Alice", email: "alice@example.com", age: 30 }

11. Automatic Semicolon Handling

Flux does not use semicolons. Statements are terminated by newlines. The lexer automatically determines statement boundaries using the following rule:

A newline terminates a statement if the last token before the newline is one of: - An identifier - A literal (integer, float, string, char, bool, none) - break, continue, return, yield - ) - ] - } - ? - ++ / -- (if ever added)

A newline does not terminate a statement if the last token before the newline is one of: - An operator (+, -, *, /, =, ==, ->, =>, |>, etc.) - An opening delimiter ({, (, [) - A comma (,) - A colon (:) - A keyword that expects a continuation (if, else, for, while, fn, etc.)

This means multi-line expressions work naturally:

let total = price
    + tax
    + shipping    // Continuation: line ends with `+`

let result = some_function(
    arg1,
    arg2,
    arg3,         // Continuation: inside parentheses
)

12. Lexer Error Handling

The lexer reports errors with precise source locations and helpful messages:

Error Example Message
Invalid character let x = § error: unexpected character '§' (U+00A7)
Unterminated string let s = "hello error: unterminated string literal
Unterminated block comment /* comment error: unterminated block comment — did you forget '*/'?
Invalid escape sequence "\q" error: invalid escape sequence '\q' — valid escapes are: \\, \n, \r, \t, \0, \", \', \u{...}
Invalid Unicode escape "\u{ZZZZ}" error: invalid Unicode escape — expected hex digits
Invalid number 0x error: incomplete hexadecimal literal
Keyword as identifier let fn = 5 error: 'fn' is a keyword and cannot be used as an identifier

The lexer continues after errors to report multiple issues in a single compilation pass (error recovery).
