Migration Guide — From Other Languages to Flux

This guide helps developers coming from other languages transition to Flux by showing equivalent patterns side by side.

From Python to Flux

Concept Python Flux
Variable x = 42 let x = 42
Mutable variable x = 42; x = 43 mut x = 42; x = 43
Type annotation x: int = 42 let x: Int = 42
String format f"Hello {name}" "Hello {name}"
Function def add(a, b): return a + b fn add(a: Int, b: Int) -> Int { return a + b }
List [1, 2, 3] [1, 2, 3]
Dictionary {"a": 1} {"a": 1}
For loop for i in range(10): for i in 0..10 {
None check if x is None: if x is None { or match x { None -> ... }
Class class Foo: class Foo { or struct Foo {
Import from os import path use flux.os.path
Lambda lambda x: x * 2 (x) => x * 2
List comprehension [x*2 for x in items] items.map((x) => x * 2)
Exception try: ... except: try { ... } catch { ... }
Async async def f(): await g() async fn f() { await g() }

Key Differences

  • Flux is statically typed — types are checked at compile time
  • Flux uses curly braces instead of indentation for blocks
  • No self parameter in methods — use self keyword directly
  • Flux variables are immutable by default — use mut for mutation
  • Flux has no None by default — use Option<T> explicitly

From Rust to Flux

Concept Rust Flux
Variable let x = 42; let x = 42
Mutable let mut x = 42; mut x = 42
String format format!("Hello {name}") "Hello {name}"
Function fn add(a: i32, b: i32) -> i32 { a + b } fn add(a: Int, b: Int) -> Int { return a + b }
Struct struct Foo { x: i32 } struct Foo { x: Int }
Enum enum Foo { A, B(i32) } enum Foo { A, B(Int) }
Pattern match match x { ... } match x { ... }
Error propagation file.read()? file.read()?
Trait trait Foo { fn bar(&self); } trait Foo { fn bar(self) }
Impl impl Foo for Bar { ... } impl Foo for Bar { ... }
Generic fn foo<T: Clone>(x: T) fn foo<T: Clone>(x: T)
Closure \|x\| x * 2 (x) => x * 2
Semicolons Required Not used
Implicit return Last expression Explicit return required
Logical AND && and
Logical OR \|\| or
Logical NOT ! not
String type String / &str String (unified)
Integer types i32, u64, etc. Int32, UInt64, etc.
Classes Not available class keyword with inheritance
ARC Rc<T> / Arc<T> (manual) Automatic for classes
Macros macro_rules! macro name!() { }

Key Differences

  • Flux uses return keyword explicitly (no implicit last-expression return in blocks)
  • No semicolons in Flux
  • English logical operators (and, or, not) instead of symbols
  • Unified String type (no &str vs String split)
  • Classes with single inheritance (in addition to structs)
  • ARC is automatic for class types
  • Simpler lifetime elision — annotations rarely needed

From Go to Flux

Concept Go Flux
Variable x := 42 let x = 42
Typed variable var x int = 42 let x: Int = 42
Constant const x = 42 const X = 42
Function func add(a, b int) int { return a + b } fn add(a: Int, b: Int) -> Int { return a + b }
Struct type Foo struct { X int } struct Foo { x: Int }
Method func (f Foo) Bar() {} impl Foo { fn bar(self) {} }
Interface type Foo interface { Bar() } interface Foo { fn bar(self) }
Error err error Result<T, E>
Error check if err != nil { return err } value? (or match)
Nil nil none / Option<T>
Goroutine go func() {}() spawn async { }
Channel ch := make(chan int) let (tx, rx) = Channel<Int>.bounded(10)
Map map[string]int{} HashMap<String, Int>.new()
Slice []int{1, 2, 3} [1, 2, 3]
For for i := 0; i < 10; i++ for i in 0..10 {
Switch switch x { case 1: } match x { 1 -> ... }
Import import "fmt" use flux.io
Public Capitalized name pub keyword
Package package main File/directory-based modules

Key Differences

  • Flux has generics with trait bounds (Go's are more limited)
  • Flux has pattern matching with exhaustiveness checking
  • Flux enforces null safety through Option<T> — no nil panics
  • Error handling uses Result<T, E> instead of multi-return (value, error)
  • Flux uses pub keyword for visibility instead of capitalization
  • Flux has classes with inheritance (Go relies on composition)

From TypeScript to Flux

Concept TypeScript Flux
Variable const x = 42 let x = 42
Mutable let x = 42 mut x = 42
Function function add(a: number, b: number): number fn add(a: Int, b: Int) -> Int
Arrow function (x) => x * 2 (x) => x * 2
Interface interface Foo { bar(): void } interface Foo { fn bar(self) }
Class class Foo { constructor() {} } class Foo { init() {} }
Generic function foo<T>(x: T): T fn foo<T>(x: T) -> T
Optional x?: string x: String? or x: Option<String>
Null check x ?? 'default' x ?? "default"
Optional chain x?.y?.z x?.y?.z
Template literal `Hello ${name}` "Hello {name}"
Array [1, 2, 3] [1, 2, 3]
Map new Map() HashMap.new()
Enum enum Color { Red, Green } enum Color { Red, Green }
Async/await async function f() { await g() } async fn f() { await g() }
Import import { foo } from 'bar' use bar.foo
Spread [...a, ...b] [...a, ...b]
Destructuring const { x, y } = point let { x, y } = point

Key Differences

  • Flux compiles to native code — not transpiled to JavaScript
  • Flux has ownership and borrowing for memory safety
  • Flux has pattern matching with exhaustiveness
  • No null or undefined — only Option<T>
  • Flux is much faster — native performance
  • Flux types are sound — no any escape hatch (except opted-in)

From Swift to Flux

Concept Swift Flux
Variable let x = 42 let x = 42
Mutable var x = 42 mut x = 42
Function func add(_ a: Int, _ b: Int) -> Int fn add(a: Int, b: Int) -> Int
Struct struct Foo { var x: Int } struct Foo { x: Int }
Class class Foo { init() {} } class Foo { init() {} }
Enum enum Foo { case bar(Int) } enum Foo { Bar(Int) }
Protocol protocol Foo { func bar() } trait Foo { fn bar(self) }
Optional var x: Int? let x: Int? or let x: Option<Int>
Unwrap x! x.unwrap()
Optional chaining x?.y x?.y
Nil coalescing x ?? 0 x ?? 0
Guard guard let x = y else { return } if let Some(x) = y { } else { return }
Switch switch x { case .a: break } match x { A -> ... }
String interp "\(name)" "{name}"
Closure { (x: Int) -> Int in x * 2 } (x: Int) => x * 2
async/await async { await f() } async { await f() }

Key Differences

  • Flux uses fn not func
  • Flux uses mut not var
  • Flux uses trait not protocol
  • Flux has ownership and borrowing in addition to ARC
  • Flux uses match with -> arrow (not switch/case)
  • Flux has {name} interpolation (not \(name))

From Kotlin to Flux

Concept Kotlin Flux
Variable val x = 42 let x = 42
Mutable var x = 42 mut x = 42
Function fun add(a: Int, b: Int): Int fn add(a: Int, b: Int) -> Int
Lambda { x -> x * 2 } (x) => x * 2
Class class Foo(val x: Int) class Foo { x: Int; init(x: Int) { ... } }
Data class data class Foo(val x: Int) #[derive(Debug, Eq, Clone)] struct Foo { x: Int }
Sealed class sealed class Result enum Result { Ok(T), Err(E) }
Null safety val x: Int? = null let x: Option<Int> = None
Safe call x?.y x?.y
Elvis x ?: 0 x ?? 0
When when(x) { 1 -> ... } match x { 1 -> ... }
Coroutine suspend fun f() {} async fn f() {}
Extension fun String.foo() {} extend String { fn foo(self) {} }
String template "Hello $name" "Hello {name}"
Companion companion object {} Static methods / module-level functions

Key Differences

  • Flux uses fn not fun
  • Flux compiles to native code (not JVM)
  • Flux uses Option<T> instead of nullable types
  • Flux has ownership and borrowing for memory safety
  • Flux uses match (not when)
  • No implicit it parameter in lambdas
