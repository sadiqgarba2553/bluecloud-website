Flux Style Guide

This document defines the official coding conventions for Flux. The flux format tool enforces formatting rules automatically, and flux lint checks for style issues.

1. Naming Conventions

Entity Convention Example
Local variables snake_case user_count, max_retries
Functions snake_case get_user, parse_config
Methods snake_case to_string, is_empty
Types (struct, class, enum, trait) PascalCase HttpClient, UserProfile
Enum variants PascalCase Some, NotFound, ConnectionError
Constants SCREAMING_SNAKE_CASE MAX_RETRIES, DEFAULT_PORT
Modules snake_case flux.http, flux.collections
Packages snake_case my_library, flux_http
Type parameters PascalCase or single uppercase T, K, V, Item, Error
Lifetimes 'lowercase 'a, 'src, 'input
File names snake_case.flux http_client.flux, user_model.flux

1.1 Prefixes and Suffixes

Pattern Convention Example
Boolean getters is_, has_, can_, should_ is_empty(), has_permission()
Conversion to_, from_, into_ to_string(), from_bytes()
Unsafe getters _unchecked suffix get_unchecked()
Builder setters with_ prefix with_timeout(), with_header()
Iteration iter, iter_mut, into_iter items.iter()
Fallible operations Use Result return type parse(), connect()

2. Formatting

2.1 Indentation

  • Use 4 spaces per indentation level
  • Never use tabs
  • flux format handles this automatically

2.2 Line Length

  • Maximum 100 characters per line
  • Break long lines at logical points:
// Good
let result = long_function_name(
    first_argument,
    second_argument,
    third_argument,
)

// Good
if user.is_authenticated()
    and user.has_permission("admin")
    and not user.is_suspended() {
    grant_access()
}

2.3 Braces

  • Opening brace on the same line as the declaration
  • Closing brace on its own line
  • Always use braces for control flow, even single-line bodies
// Good
fn process(data: Data) -> Result<Output, Error> {
    if data.is_empty() {
        return Err(Error.EmptyInput)
    }
    return Ok(transform(data))
}

// Bad — no braces on single-line if
// if data.is_empty() return Err(Error.EmptyInput)

2.4 Blank Lines

  • Two blank lines between top-level items (functions, structs, impls)
  • One blank line between logical sections within a function
  • No blank line after opening brace or before closing brace
struct User {
    name: String
    age: Int
}


impl User {
    fn new(name: String, age: Int) -> User {
        return User { name: name, age: age }
    }

    fn greet(self) -> String {
        return "Hello, {self.name}!"
    }
}


fn main() {
    let user = User.new("Alice", 30)
    print(user.greet())
}

2.5 Trailing Commas

Use trailing commas in multi-line lists:

let items = [
    "first",
    "second",
    "third",   // trailing comma
]

fn create(
    name: String,
    age: Int,
    email: String,   // trailing comma
) -> User {
    // ...
}

2.6 Import Ordering

Imports should be grouped and sorted:

// 1. Standard library
use flux.io.{print, read_line}
use flux.collections.HashMap

// 2. Third-party packages
use serde.{Serialize, Deserialize}
use tokio.runtime.Runtime

// 3. Local modules
use self.models.User
use self.services.auth

3. Documentation

3.1 Doc Comments

All public items must have documentation comments:

/// Creates a new HTTP client with the given configuration.
///
/// # Parameters
/// - `config` — Client configuration (timeouts, headers, etc.)
///
/// # Returns
/// A configured `HttpClient` ready to make requests.
///
/// # Examples
/// ```flux
/// let client = HttpClient.new(Config.default())
/// let response = await client.get("https://example.com")
/// ```
pub fn new(config: Config) -> HttpClient {
    // ...
}

3.2 Section Headers in Doc Comments

Section Purpose
(opening paragraph) Brief description
# Parameters Describe each parameter
# Returns Describe the return value
# Errors Document when errors are returned
# Panics Document when the function panics
# Examples Working code examples
# Safety Safety requirements for unsafe functions

4. Code Patterns

4.1 Prefer let Over mut

// Good — compute the final value directly
let status = if score >= 90 { "A" } else { "B" }

// Avoid — unnecessary mutation
mut status = ""
if score >= 90 {
    status = "A"
} else {
    status = "B"
}

4.2 Prefer match Over if-else Chains

// Good
match status_code {
    200 -> handle_ok(response)
    404 -> handle_not_found()
    500 -> handle_server_error()
    _   -> handle_unknown(status_code)
}

// Avoid
if status_code == 200 {
    handle_ok(response)
} else if status_code == 404 {
    handle_not_found()
} else if status_code == 500 {
    handle_server_error()
} else {
    handle_unknown(status_code)
}

4.3 Use Early Returns

// Good — early returns reduce nesting
fn process(data: Data) -> Result<Output, Error> {
    if data.is_empty() {
        return Err(Error.EmptyInput)
    }

    if not data.is_valid() {
        return Err(Error.InvalidData)
    }

    let result = transform(data)
    return Ok(result)
}

4.4 Use the ? Operator

// Good
fn load_config(path: String) -> Result<Config, AppError> {
    let content = File.read(path)?
    let config = json.parse(content)?
    return Ok(config)
}

// Avoid — verbose match chains
fn load_config_verbose(path: String) -> Result<Config, AppError> {
    let content = match File.read(path) {
        Ok(c) -> c
        Err(e) -> return Err(AppError.from(e))
    }
    // ...
}

5. Anti-Patterns

5.1 Don't Use unwrap() in Production Code

// Bad — panics at runtime
let value = risky_operation().unwrap()

// Good — handle the error
let value = match risky_operation() {
    Ok(v)  -> v
    Err(e) -> {
        log.error("Operation failed: {e}")
        return default_value
    }
}

// Acceptable — when you have a logical guarantee
let first = non_empty_list.first().unwrap()  // Only if you KNOW it's non-empty
// Better:
let first = non_empty_list.first().expect("list was verified non-empty above")

5.2 Don't Overuse any

// Bad
fn process(data: any) -> any { ... }

// Good
fn process<T: Processable>(data: T) -> ProcessResult<T> { ... }

5.3 Don't Ignore Errors

// Bad — silently ignores errors
let _ = save_file()

// Good — handle or explicitly acknowledge
match save_file() {
    Ok(()) -> {}
    Err(e) -> log.warn("Could not save: {e}")
}

// Acceptable — when you truly don't care (use a comment)
let _ = save_file()  // Best-effort save, failure is OK

6. File Organization

//! Module-level documentation goes first.

// Imports (grouped and sorted)
use flux.io.print
use flux.collections.HashMap

// Constants
const MAX_SIZE = 1024

// Type aliases
type StringMap<V> = HashMap<String, V>

// Types (structs, enums, classes)
pub struct MyType {
    // fields...
}

// Trait implementations
impl Display for MyType {
    // ...
}

// Methods
impl MyType {
    // Public methods first, then private
    pub fn new() -> MyType { ... }
    pub fn process(self) { ... }

    fn internal_helper(self) { ... }
}

// Free functions
pub fn utility_function() { ... }

// Tests at the bottom
#[test]
fn test_my_type() { ... }
